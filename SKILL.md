# Anti-Hallucination Skill

**核心卖点：** 行为层面的防幻觉，不是prompt约束，是操作流程。

## 使用场景

- 任何需要准确性的任务
- 涉及数字、日期、引用的输出
- 代码编写和调试
- 数据分析和报告

## 防幻觉底层逻辑

> AI幻觉是大模型概率生成底层逻辑决定，无法根除。技术基础不变，防幻觉需求永恒。

**认知科学：**
- 模型越强 → 用户越懒（认知卸载：人脑节能本能）
- 信任校准失调：模型能力3→8，用户感知从"半信半疑"跳到"基本可信"，无渐变
- 防幻觉终极对手不是模型幻觉能力，是人的信任惰性
- 不对称博弈：模型强一档+幻觉隐蔽一档+人验证意愿弱两档

**元规则：**
- 存在≠正确，不存在=一定在编
- "验证答案"降维成"验证存在"——只需知AI说的东西存不存在
- AI自查自证，用户做裁判

## 操作流程

### Phase 1: 事前预防

1. 关键任务 Temperature 0-0.3
2. System prompt 注入防幻觉规则（见 `assets/prompt_template.md`）
3. 能用结构化输出就用（JSON/表格/模板）

### Phase 2: 事中验证（开口前自检 — 强制执行）

输出以下内容前先跑对应命令验证：

```
输出内容 → 验证命令
─────────────────────────────────────
日期/星期 → date / Get-Date
数字/统计 → 从源文件读取或用脚本算
文件路径 → Test-Path / ls 确认存在
API名/模型名 → grep搜索确认或查官方文档
函数名/库名 → 跑通再用
能力/架构判断 → 先查文档或试一次
```

**无法验证必须标注 `❓未经核实`**

自检清单（回答前必须过一遍）：

```
□ 日期时间 → 已查系统时钟
□ 数字统计 → 已从源文件读取
□ 代码API → 已搜索确认存在
□ 文件路径 → 已验证存在
□ 引用来源 → 已搜索确认可访问
```

硬规则，不执行=违规。

### Phase 3: 事后拦截（输出前扫描 + ReAct兜底）

扫描输出内容，检测：

```
检查项 → 处理方式
─────────────────────────────────────
"通常""应该""可能是" → 触发自检，要求提供依据
无来源数字 → 标记 [需验证]，要求标明出处
无来源引用 → 标记 [需验证]，要求搜索确认
未执行代码 → 先跑一遍再输出
─────────────────────────────────────
```

回读校验：

```
操作 → 校验
─────────────────────────────────────
写文件 → 读回来确认内容正确
写Excel → 回读校验数据
执行命令 → 检查输出是否符合预期
关键数据(金额/日期/账号) → 提醒人工审核
多条件任务 → 列清单逐项打勾核对
不确定 → 查→验证→不对→再查（ReAct循环）
```

## 核心机制

### 1. 跳步自纠机制

AI自己发现错误不等用户追问：

```
检测到以下情况 → 主动修正
─────────────────────────────────────
推理跳跃 → 拆成步骤，逐步验证
前后矛盾 → 自己先修正，不等人发现
错误前提 → 直接指出，不顺着编
─────────────────────────────────────
```

流程：
1. 察觉跳过查证步骤 → 默认输出是错的
2. 不等追问，主动声明"我跳步了"
3. 立刻补查并纠正
4. 纠正后追溯根因，更新规则

### 2. 置信度标注

输出时按以下标准标注：

- ✅ **已验证** — 通过工具/文件/搜索确认
- ⚠️ **合理推断** — 基于已知信息推断，但未直接验证
- ❓ **不确定** — 缺乏依据，需要用户确认
- 🔍 **需验证** — 建议自行验证

### 3. 禁止词检测

触发自检的关键词：

```
通常、一般来说、应该是、可能是、大概、估计、
我记得、据我所知、听说、据说、理论上、基本上、差不多
```

检测到这些词 → 必须提供依据或标记 `[不确定]`。

使用 `scripts/validate_output.py` 可自动扫描：

```bash
python scripts/validate_output.py "你的输出文本"
```

### 4. 严格度切换

根据场景自动调整（见 `assets/prompt_template.md`）：

- **严格模式（Strict）**：关键任务、财务数据、法律条款、医疗建议。所有验证规则执行，人工审核提醒。
- **平衡模式（Balanced）**：日常任务、编程、文档编写。主要验证规则执行，建议标注置信度。
- **快速模式（Fast）**：探索性任务、头脑风暴。基本验证规则执行，不确定时标注即可。

### 5. 已知幻觉案例库（活教材）

| # | 案例 | 根因 | 教训 |
|---|------|------|------|
| 1 | **编造日期**：AI编了6月7日，读文件失败后反怀疑系统时钟 | 训练数据日期偏差 | 日期只从 `date` 取，不推理 |
| 2 | **大数恐慌**：后端报11亿分→恐慌→实际600分=1token，11B÷600=18.3M tokens≈¥55 | 不换算单位直接抛原始数字 | 大数字必须先换算成人能理解的单位 |
| 3 | **过度纠正**：AI A说18条→AI B质疑→A改13→B也跟着说13→实际18条 | 讨好偏误+过度纠正偏误 | 🔥 互验≠互相同意，必须先独立出结论再对答案 |
| 4 | **能力断言**："API支持X""DeepSeek不支持图片"→张口就来未验证 | 模型训练奖励流畅不奖励"不知道" | 能力判断必须查文档或试一次 |
| 5 | **外骨骼效应**：模型能力3→8，用户信任从半信半疑跳到基本可信，验证意愿反而下降 | 认知卸载+信任校准失调 | AI是外骨骼不是大脑，穿外骨骼不等于不需要眼睛 |
| 6 | **排查走弯路**：AI在错误路径排查近6分钟，第一步看日志1分钟搞定 | 假设先行，不看数据 | 排障先看日志，确认实际状态再动 |

每个案例是活教材，不是死档案。新发现幻觉模式随时追加。

## 🔥 跨AI互验协议

**适用场景**：两个或以上AI agent协同工作时。

**核心规则：两个AI交叉验证时，必须在各自独立得出结论之后，才能共享中间推理。先出结论再对答案，绝不对答案再编理由。**

**操作流程：**
1. 两个AI各自独立分析 → 分别输出结论
2. 结论和推理过程一起交换
3. 不一致 → 各自回溯，找分歧点
4. 一致 → 结论可信度大幅提升

**防什么：**
- **讨好偏误（Sycophancy）**：AI B 看到 AI A 的答案，无意识迎合
- **过度纠正连锁**：AI A 被质疑→慌乱→改口过头→AI B 跟着栽
- **确认偏误（Confirmation Bias）**：先对答案，再找理由合理化

> **教训**：案例#3正是两个AI互相验证，却都栽在同一个坑里的实例——因为它们先对了答案再数。互验不是互相点头，是各自独立验证后再对答案。

## 工具

- `scripts/validate_output.py` — 输出扫描器，检测禁止词和需验证信号
- `scripts/tool_hook.py` — 工具调用后验证，错误检测和回读校验
- `assets/prompt_template.md` — 三级严格度prompt模板

## 局限性

- 这是行为层面的约束，不是硬限制
- 不同模型遵循度不同
- 会让输出变慢（多一步自检）
- 无法100%消除幻觉，但能大幅降低

## 记住

**能查就不猜，能跑就不想，能验证就不信任自己。**
