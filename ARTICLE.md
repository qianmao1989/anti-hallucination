# 我让5个AI互相取长补短，造了一个防幻觉工具

## 1. 起点：我为什么受不了AI编东西

我做百战CD管理的时候，AI算错了一个数字。不是差一点点，是整个包四周的计算全废了。我花了两个小时排查，最后发现是AI把一个百分比"合理推测"错了。

还有一次写代码，AI信誓旦旦地告诉我某个API的用法，我照着写完一跑，报错。查了半天，那个API根本不存在，是AI编出来的。

最让我受不了的不是"不知道"。不知道就说不知道，我来查。最受不了的是"自信地胡说"——语气笃定，逻辑通顺，但内容是假的。

市面上防幻觉的方案我看了不少，基本都是一个思路：在prompt里加一句"请不要编造信息"或者"如果你不确定请说不知道"。这叫prompt约束，是软的。AI心情好的时候遵守，心情不好的时候照编不误。

我的想法不一样。不是告诉AI"不要编"，而是给它一套操作流程，让它"没法编"。能查就不猜，能跑就不想，能验证就不信任自己。这不是约束，是行为层面的改变。

## 2. 方法：飞书统一发问，人工拍板合并

我有5个AI可以用。不是五个聊天窗口，是五个不同定位的AI工具：

| AI | 角色定位 | 擅长什么 |
|----|----------|----------|
| CC（Claude Code） | 架构师 | 代码实现、规则骨架 |
| 小助理（OpenClaw） | 工程师 | 脚本、模板、文档整合 |
| QQbot | 产品经理 | 用户体验、置信度标注 |
| DeepSeek | 战略顾问 | 定位区分、差异化 |
| 海马士（Hermes） | 质检员 | 事实核查、验证机制 |

我的做法是：在飞书统一发一个问题——"你怎么看防幻觉？防什么？怎么防？"——然后把每个AI的回答分别收集起来，汇总转发，取长补短。

为什么不搞群聊？因为群聊会互相影响。强势的AI一带节奏，其他的就跟着附和了。单独发问，每个AI只能拿出自己的"真话"，没有别人的观点可以参考。

最终谁来拍板？我。AI出方案，人做判断。这个"人作为协调者"的模式，比"AI自己开会"有价值得多。

## 3. 各AI的独特贡献

### CC（Claude Code）：骨架搭建者

CC干的是搭骨架的活。它产出了5个文件：

- `rules/pre-check.md`——开口前自检规则，日期、数字、代码、路径、引用，每类都有"触发词→执行动作"的映射
- `rules/in-process.md`——事中验证规则，核心就四句话：能查不猜，能跑不想，交叉验证，思维链拆解
- `rules/post-check.md`——事后拦截规则，回读校验、人工审核提醒、ReAct循环兜底
- `hooks/fact-check.md`——事实核查钩子，5类模式（日期/数字/代码/路径/引用），每类带Python伪代码
- `hooks/code-verify.md`——代码验证钩子，语法/依赖/API/执行四级验证

CC最独到的观点是"跳步自纠"：AI自己发现错误不等用户追问。不是等你问"这个对吗？"，而是AI自己发现推理跳了一步，主动声明"我跳步了"然后补查。

没有CC的骨架，其他人的贡献都是散装零件。

### 小助理（OpenClaw）：工程实现者

小助理干的是把规则变成工具。它产出了3个文件：

- `scripts/validate_output.py`——输出扫描器，18个禁止词（中英双语）+ 8类正则模式，能自动检测日期、百分比、金额、URL、函数调用、文件路径、版本号、API端点
- `scripts/tool_hook.py`——工具调用后验证，空输出检测、错误信号检测、文件写入回读验证
- `assets/prompt_template.md`——三级严格度模板，Strict/Balanced/Fast各一套system prompt，附动态模式选择表

小助理最独到的观点是"三级严格度"：不是所有场景都需要同样的防幻觉强度。写财务报表用Strict，日常聊天用Fast，大多数情况用Balanced。

CC的规则是"告诉AI怎么做"，小助理的脚本是"让工具帮你检查"。两层互补。

### QQbot：用户体验设计师

QQbot不写代码，它关心的是"怎么让用户信任AI的输出"。

它的核心贡献是置信度标注系统：输出要分层次，不能一锅端。

- [确定] — 有明确来源，已验证
- [推断] — 基于已知信息合理推断
- [不确定] — 需要用户确认
- [需验证] — 建议用户自行验证

这套标注被写进了规则文件和prompt模板。有了它，用户一眼就能看出哪些内容可信、哪些需要自己再查一遍。

### DeepSeek：战略定位师

DeepSeek说了一句话，把整个项目的定位拎清了：

> "大部分防幻觉方案是prompt层面的，你这个是行为层面的——不是告诉AI'不要编'，而是给它一套具体的操作流程。"

在DeepSeek之前，这个skill可能只是"又一个prompt模板"。有了这个区分，它变成了"操作流程"。这是整个项目的转折点。

一句话定位，价值千金。

### 海马士（Hermes）：质检机制设计者

海马士的核心贡献是事实核查钩子的概念：检测到数字、日期、引用，自动触发验证，不是等用户发现。

它还提供了视觉校验的视角——这个能力在其他四个AI中是没有的。在事后拦截环节，海马士补上了关键的拼图：怎么让验证结果一目了然。

## 4. 合体：怎么把5份答案变成1个工具

5个AI的回答收齐了，接下来是我的活。

**第一步：提取各自独特观点。** CC重骨架、小助理重工具、QQbot重体验、DeepSeek重定位、海马士重流程。每个人的侧重点不同，这正是我要的。

**第二步：找共性。** 大家都提到了"验证""回读""不确定就说不确定"。这些是共识，直接写进核心规则。

**第三步：处理冲突。** 这是最关键的——

冲突1：严格度该固定还是可调？CC的规则是固定的，没有Strict/Balanced/Fast的区分。小助理提出三级可调。我拍板：用小助理的方案。CC的规则作为Strict模式的实现，Balanced和Fast是它的降级版。理由很简单：不同场景需要不同强度，一刀切不现实。

冲突2：置信度用文字还是emoji？QQbot用[确定][推断][不确定]，小助理用✅⚠️❓。我拍板：两套都保留。文字版放在rules/里（正式场景），emoji版放在prompt_template.md里（快速识别场景）。

**第四步：分工合并。** 结构归CC，执行归小助理，体验归QQbot，定位归DeepSeek，流程归海马士。SKILL.md的最终合并由小助理负责——它最擅长文档整合。

最终SKILL.md有261行，中英双语，包含四大机制：

1. **事实核查钩子** ← 海马士 + CC
2. **输出拦截器** ← 小助理 + QQbot
3. **跳步自纠** ← CC独创
4. **回读校验** ← CC + 小助理

## 5. 成果：anti-hallucination skill v1.0.0

最终产品长这样：

```
anti-hallucination/
├── SKILL.md                    # 核心规则文档（261行，中英双语）
├── _meta.json                  # 元数据
├── rules/                      # 详细规则
│   ├── pre-check.md            # 开口前自检
│   ├── in-process.md           # 事中验证
│   └── post-check.md           # 事后拦截
├── hooks/                      # 钩子机制
│   ├── fact-check.md           # 事实核查钩子
│   └── code-verify.md          # 代码验证钩子
├── scripts/                    # 可执行脚本
│   ├── validate_output.py      # 输出扫描器
│   └── tool_hook.py            # 工具调用后验证
└── assets/                     # 模板
    └── prompt_template.md      # 三级严格度模板
```

**实测效果：**

测试1，禁止词检测：
```
输入："这个API应该是可以的，通常情况下会返回200。我记得上次测试过，大概是30ms响应时间。"
输出：⚠️ 检测到6个禁止词（我记得、应该、通常、大概、应该是、大概是）
```

测试2，信号检测：
```
输入："用户增长了30%，从1000人增加到1300人。API端点是/api/v1/users，版本号v2.1.3。文件路径是D:\data\users.csv。"
输出：🔍 需要验证的信号：30%（百分比）、/api/v1/users（文件路径）、v2.1.3（版本号）
```

一次扫描就能把AI输出里的"水分"标出来，不用人工逐句检查。

## 6. 反思：这种协作方式的优劣

**优势：**
- 视角多元。5个AI各有专长，CC看架构、小助理看实现、QQbot看体验、DeepSeek看定位、海马士看流程。单一AI很难同时兼顾这五个维度。
- 互补性强。每个人的短板恰好是别人的长板。
- 人做裁判。我拍板合并，避免了AI互相附和或"投票表决"的陷阱。
- 冲突带来深度。CC和小助理在严格度上的冲突，最终产出了比两者都更好的三级方案。

**劣势：**
- 协调成本。需要有人来合并和取舍，这个人必须对全局有判断力。
- 一致性维护。5个AI的风格不同，合并后需要统一语言和格式。
- 信息损耗。单独发问可能遗漏交互中碰撞出的灵感。

**适合什么场景：**
- 需要多角度思考的问题
- 需要专业分工的任务
- 需要避免"群体思维"的决策

**不适合什么场景：**
- 简单任务（杀鸡用牛刀）
- 紧急任务（协调太慢）
- 需要实时交互的场景

## 7. 彩蛋：对其他开发者的建议

**如果你想复制这种协作方式：**

1. 找3-5个不同定位的AI。不要找5个同类型的，那等于问同一个人5遍。
2. 在一个平台统一发问。飞书、微信、任何聊天工具都行，关键是同一个问题分别发给每个AI。
3. 收集完再合并。重点是"取长补短"，不是"投票表决"。多数AI同意不代表正确，少数派的观点可能更有价值。
4. **你**做最终判断。不要让AI自己决定用谁的方案。AI没有全局视角，你有。

**如果你想用这个skill：**
- GitHub已开源，MIT协议，欢迎贡献
- 三级严格度可根据场景调整：Strict（财务/法律/关键数据）、Balanced（日常任务）、Fast（探索/头脑风暴）
- 欢迎反馈使用效果

---

*这个skill不是5个AI自动产出的。它是5个AI各自出主意、一个人拍板合并的结果。AI是工具，人是决策者。*
